# Creating a finding model

Use `scripts/finding_authoring/create_model.py` for all new model creation. It allocates unique OIFM and OIFMA IDs, applies standard codes, and writes canonical JSON.

## Before calling the script

Apply the rules in `naming.md`, `synonym_rules.md`, and `scope_and_specificity.md` to your inputs:

- Expand acronyms in the canonical name; keep the acronym as a synonym.
- Replace brand names with generic terms; keep the brand as a synonym.
- Minimize eponyms where a descriptive term is standard; keep the eponym as a synonym.
- Verify each synonym is truly synonymous — not a subtype, supertype, or different finding.
- Scope broad names (`mass` → `pulmonary mass`, etc.); keep the unscoped short form as a synonym.
- All names lowercase unless capitalization is justified.

If the finding has no meaningful synonyms, **omit the `--synonyms` flag entirely**. The underlying schema requires at least one synonym when the list is provided.

## Single model

```bash
uv run --env-file .env scripts/finding_authoring/create_model.py \
    --name "<finding name>" \
    --description "<one- to two-sentence clinical description>" \
    --synonyms "synonym1" "synonym2" \
    --tags <tag1> <tag2> <tag3> \
    --source <source_code> \
    --contributor <person_key> <org_key> \
    --cfp-pairs larger-smaller \
    --output defs/<filename>.fm.json
```

Output: `filepath|name|oifm_id` on one line on success.

### `--cfp-pairs`: direction-of-change values on `change from prior`

By default the script emits only the **minimum set** (`unchanged, stable, new, resolved`). Opt in to direction pairs via `--cfp-pairs <csv>`. Valid pair keys:

- `larger-smaller` — masses, effusions, discrete lesions with measurable size
- `increased-decreased` — quantities, density, extent
- `worsened-improved` — conditions, disease states, chronic processes

Combine as needed: `--cfp-pairs larger-smaller,worsened-improved`. Omit entirely for findings where no direction is clinically natural (anatomic variants, devices, congenital structures). See `presence_and_change.md` for per-finding guidance.

To determine the filename ahead of time:

```bash
uv run --env-file .env --with findingmodel --with openai python -c "from findingmodel.common import model_file_name; print(model_file_name('<finding name>'))"
```

## Batch (JSON on stdin)

```bash
uv run --env-file .env scripts/finding_authoring/create_model.py --batch << 'EOF'
{
    "source": "<source_code>",
    "contributors": ["<person_key>", "<org_key>"],
    "tags": ["<tag1>", "<tag2>", "<tag3>"],
    "cfp_pairs": "larger-smaller",
    "models": [
        {
            "name": "finding one",
            "description": "Description of finding one.",
            "synonyms": ["synonym a", "synonym b"]
        },
        {
            "name": "finding two",
            "description": "Description of finding two.",
            "cfp_pairs": "worsened-improved"
        },
        {
            "name": "finding three",
            "description": "An anatomic variant; no direction pairs.",
            "cfp_pairs": ""
        }
    ]
}
EOF
```

Output: one `filepath|name|oifm_id` line per successfully created model. Errors are printed to stderr as `ERROR creating '<name>': <reason>`.

Per-model overrides: each entry in `models` may set its own `tags`, `output`, or `cfp_pairs` to override the batch-level defaults. Set `cfp_pairs: ""` to force the minimum set on a single model when the batch-level default adds pairs.

## What the script does NOT do

- It does **not** decide whether the finding should exist — run the search/triage step first (`search_and_triage.md`).
- It does **not** verify your synonyms are correct — the quality review step (`quality_checklist.md`) does.
- It does **not** winnow direction-of-change values in `change from prior` — quality review does (`presence_and_change.md`).
- It does **not** run the repo validator — `scripts/validator.py` runs separately.

## Shell tip

For anything multi-line — including the batch JSON config and any inline `python -c` snippet longer than one short line — use a heredoc (`<< 'EOF' … EOF` or `<< 'PYEOF' … PYEOF`) instead of quoting. Heredocs avoid shell-escaping landmines around quotes, backslashes, and newlines.

## Critical constraints

- **Never generate OIFM or OIFMA IDs manually.** The script allocates them and checks uniqueness against the index.
- **Never edit generated files** `text/*.md`, `index.md`, `ids.json` — they are regenerated by `scripts/validator.py`.
- **Do not run `scripts/validator.py` from inside this workflow.** It is run separately, outside the authoring loop.
- **Do not commit** without explicit user permission.
